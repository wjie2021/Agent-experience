# Claude Code Skill 使用分享

## 一、自己对AI工具的使用历程

其实在早期，chatgpt等各种LLM对话工具出现的时候我其实基本上是吧AI当成搜索引擎使用的，出现大模型的聊天工具后，我就从从面向csdn编程到面向Deepseek，chatgpt编程。很长的时间都停留在这个层面，或者最多在IDEA里面进行一个代码补全，或者在ide中提需求帮忙改代码，自己选择是否采纳。
对于复杂逻辑也不太方向交给AI，所以即使AI发展的很快，由于没有深入了解，其实也没感觉的智能体的便捷与强大。

## 二、自己对智能体的理解

老板那天跟咱们演讲的时候说了skill，我当时其实对skill没有什么概念，觉得又是哪些卖token的大厂搞出来的噱头，于是自己去了解了一下Skill，从而扩展到了从LLM，到上下文，到functionCalling到mcp到skill。

我今天跟大家主要分享的就是对智能体的Skill的理解和使用。

早期对qoder的使用应该都是在ide工具的对话框中描述需求，或者贴文档，然后生成代码。这样其实不太好控制生成的代码质量，还有生成代码的格式，比如魔法值，注释风格，变量风格等，但是如果在提示词加上这些其实很麻烦，所以把这些常用的、规范化的提示词总结整理一下，放到一个指定的位置，就是skill了，可以简单理解成就是提示词换了个位置而已。

## 三、对skill的使用

先贴一下成果：
因为直接让智能体写代码不太可控，而且一个复杂的业务逻辑出来之后一行一行地复读代码其实很麻烦，我就想有没有什么能提高生成代码的质量，以及提高代码复读的方式。所以我结合自己工作流以及软件的开发流程，用了以下的3个技能来辅助我的开发。
总体而言就是从需求分析-概要设计-详细设计-编写代码-复读代码。
于是我编写了3个skill来协助我完成以上的流程：
1、req-to-api：需求→设计→API全链路规范化技能包，就是将需求文档，先分析为一个总结性的文档，包括一些业务逻辑，业务状态，可能的功能关联。然后编写建表语句，最后生成API文档。再到最后的自检过程。
2、api-to-code：让智能体阅读刚才的API文档。遵循我的开发习惯，代码规范，开始编写代码。最后再进行自检。将API接口文档转化为完整的Controller+Service+Cmd/Query/VO后端代码实现。适用于用户提供API文档（Markdown/JSON格式）并要求生成Spring Boot后端代码的场景，涵盖接口解析、代码生成、自检验证、交叉核对全流程。
3、code-review：对指定模块的源代码进行多维度质量审查，基于项目规范和最佳实践识别问题，通过置信度评分分级展示，最终生成Markdown审查报告保存到项目的docs/review目录。
然后切到IDEA进行演示和分享。

我感觉大家可以研究下skill这个东西，这个说复杂也复杂，说简单也简单。其实说白了就是将自己本来的要在输入框输入的一大推重复的提示词放到一个固定的文件夹里，这些skill其实也是自然语言描写的，可以将自己平时的工作流用文档写出来，这样就可以节省很多时间。
这个应该有个官方文档，大家可以参考下。
https://docs.qoder.com/zh/cli/Skills

## 五、其他的：记忆和规则

我用的较少，因为我觉得这些其实和skill差不多，也是提示词换了个位置，
可以在规则里可以把自己的开发规范，任务的边界放进去，比如xml的sql的对齐方式，代码中的注释风格等，都可以放进去。

## 六：最后：一点小技巧

AI工具生成的代码目前来看不是特别健壮，手写代码的过程是有思考的，是有依据的，所以让AI写代码之前先让它生成一个详细设计文档，或者设计方案，我们去核对文档，然后再让AI写代码，最最重要的就是在提示词中需要说明完成代码的编写后进行交叉核验，这样的正确率会高很多。
